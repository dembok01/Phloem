# UIUX-REVIEW.md — PHLOEM Design Audit

**Phase A of the UI/UX overhaul · 2026-07-13.** Every role was walked with the seeded
accounts at 1440×900 and 390×844; 55+ screens captured to `design-audit/before/`
(`scripts/design-audit.ts`). Demo data was advanced through the real §6 RPCs (signed in
as the seeded users — no raw writes, no schema changes) so screens show a full story:
Meera Krishnan active with four submitted reports, Rajan Pillai mid-pipeline with mixed
meeting/report states, K. V. Gopalan mid-onboarding, Padma Nair holding an open invite.

Audience order for every judgment below: (1) elderly members & adult children —
calm, legible, reassuring; (2) clinicians — speed and scanability; (3) coordinator —
density and keyboard efficiency.

---

## 1 · Global findings (affect every screen)

### G-1 — The entire app renders in fallback Times serif ⚠ root cause found
`app/globals.css:10` — `@theme inline { --font-sans: var(--font-sans); }` is
**self-referential**, so `font-family: var(--font-sans)` is an invalid circular
reference and every element falls back to the browser default serif. The Geist fonts
loaded in `app/layout.tsx` are bound to `--font-geist-sans`, which nothing consumes.
Net effect: buttons, tables, forms — all Times New Roman. The single most visible
"unstyled default" in the product, and it silently defines the app's whole look.
*(Exception: `ReportView` sets its own font stack inline, which is why reports are the
only screens that look intentional.)*

### G-2 — Achromatic token palette; the product has no color identity
Every `:root` token in `globals.css` is `oklch(x 0 0)` — zero chroma. Primary is
near-black; every chip/badge color on screens comes from ad-hoc Tailwind classes
(`bg-emerald-100`, `bg-amber-50`, `text-red-*`) rather than tokens. The only brand
color anywhere is the logo PNG. There is no role color-coding (§brief: role-colored
context), no semantic color system (success/warning/danger used inconsistently — see
G-6).

### G-3 — Browser-tab title is "Create Next App"
`app/layout.tsx:16` still carries the scaffold metadata (`title: "Create Next App"`,
description "Generated by create next app"). No per-page titles anywhere. Favicon is
the scaffold default.

### G-4 — Secondary text fails WCAG AA
`--muted-foreground: oklch(0.556 0 0)` ≈ #8a8a8a on white ≈ **3.5:1** — below the
4.5:1 AA threshold for normal text. This is the color of nearly all supporting copy:
page subtitles, table headers, empty states, card metadata, consultation details.
Elderly-facing portal uses it heavily. (AAA in elderly mode requires 7:1.)

### G-5 — Focus states are near-invisible
Global `* { outline-ring/50 }` with `--ring: oklch(0.708 0 0)` = a 50%-alpha light
grey outline. Keyboard users (the coordinator lives here) get essentially no visible
focus indication on white backgrounds.

### G-6 — Status semantics are inconsistent and sometimes dangerously wrong
- Trainer's "Doctor's clearance" tab shows **"Cleared with restrictions" in a green
  success banner** (`trainer--client-clearance.png`) — cautionary clinical content
  styled as good news; the restriction list below it is undifferentiated bullets.
- Red-flag banners are **red-on-pink with severity jargon** ("Chest pain on exertion —
  high"); §11 specifies **amber with plain language** ("…doctor review required before
  training").
- Coordinator dual chips render Meeting and Report state in the **same green pill**
  (`coordinator--member-active.png`) — meeting vs report is illegible at a glance,
  the exact thing the brief calls out.
- The same "Completed/Meeting done/Report submitted/Pending" concepts appear as
  different words and colors on coordinator, clinician, portal, and admin surfaces.

### G-7 — Touch targets & type below spec
Base `Button` is `h-8` (32px) at 14px text; segmented scale buttons in forms are
~30px tall. §11 requires ≥16px base and large touch targets; the brief requires ≥44px
targets (48px elderly). Portal base type is 18px but interactive rows are ~40px.

### G-8 — Vast dead space; no page grid
Screens are single centered columns (~768px) floating in 1440px of white
(`coordinator--today-busy.png`, `elderly--home.png`, `admin--overview.png`). The nav
tab rail spans full width while content hugs a narrow column, so the page reads as
left-heavy. No consistent page header pattern (some pages have back-links, some
don't; title/subtitle/actions land in different spots per page).

### G-9 — No feedback layer
No toast system. Server-action outcomes surface as `?ok=`/`?error=` query-param
banners on some pages, silently nowhere on others. Nothing confirms "Scheduled",
"Assigned", "Saved" with the verb of the button pressed. The Start Program moment —
the emotional peak of the coordinator flow — yields a plain text line.

### G-10 — Empty states are labels, not invitations
"Empty" (dashed box, pipeline), "No upcoming consultations scheduled.", "No packages
renewing in the next 30 days." None tells the user what to do next except the
clinician clients list. Loading skeletons exist (`loading.tsx`) but are generic grey
bars that don't match the real layout shapes.

### G-11 — Machine copy leaks everywhere
- Today queue: "nutritionist meeting — Tue, 14 Jul, 03:21 am" (lowercase role,
  passive phrasing; row's action unclear).
- Report K/V rows: raw ISO date "2026-07-12" (violates §11 human-readable dates);
  "Member caregiver" (missing "+"); "video" lowercase in schedule rows.
- Role badges in the header are lowercase enum values ("caregiver", "doctor").
- Mixed vocabulary: "Clients" (clinician), "Members" (admin), "people in your care"
  (portal) for the same concept. Standardize on **member** (clinician nav may keep
  "My members").

### G-12 — Mobile: pipeline board unusable, tables overflow
At 390px the pipeline shows 1.3 columns with no scroll affordance
(`coordinator--pipeline--mobile.png`); admin tables need horizontal scroll containers;
the wizard is serviceable but cramped. Caregiver portal mobile is passable.

## 2 · Per-surface findings

### Login & invite accept (`public--login`, `public--invite-accept`)
Bare white card on grey; scaffold-default look, serif type (G-1). No sense of arrival
or trust for a healthcare product. Invite accept is functionally good (role badge,
readonly email) but visually identical to login — no "you were invited by your
coordinator" warmth, no next-steps orientation.

### App shell (all roles)
Header = logo + bell + name + role-enum badge + Sign out. No role context color, no
current-section indicator in the header, breadcrumbs only as ad-hoc "← Pipeline"
links. The nav-tabs component underlines the active tab but inactive tabs are muted
grey (G-4 contrast). Notification badge is a red dot that clips the bell glyph.

### Caregiver portal home (`caregiver--home`)
The emotional core, currently a status readout. "Your family portal" heading, then a
card with a 3-segment grey progress bar (cycle markers unlabeled), "Cycle 1 · Day 1
of 30" in small muted type on the right. No member identity (photo/initials), no
plain-language reassurance ("on track"), no journey visualization, appointments and
care team are grey list rows. Plans/Reports/Schedule cards are icon+word only.

### Elderly mode (`elderly--home`)
Meets the letter of §10 (3 items, big type) but is three plain white cards adrift in
white space — no warmth, no confidence cues, serif fallback. Touch rows ~72px (good).
No reduced-motion enforcement (irrelevant today — nothing moves — but must be forced
once motion exists).

### Coordinator today queue (`coordinator--today-busy`)
Rows are passive statements ("Awaiting the doctor report") with a chevron; the
one-clear-action promise isn't kept (no verb, no button). Group headers "Today (1)"
/ "This week (3)" are plain bold text. No keyboard navigation, no Cmd+K, no
overdue/red-flag visual weight.

### Coordinator pipeline (`coordinator--pipeline`)
Column headers are muted 14px text + count badge; cards show name + one muted line;
the red-flag dot is the only signal that reads. Last column clips off-screen with no
scroll affordance (desktop AND mobile). "Empty" dashed boxes. No drag affordances
(spec's §10 board is click-to-manage; the brief asks for drag *affordances* —
proposal territory if actual drag needs new mutations).

### Coordinator member page (`coordinator--member-active`, `--member-mixed-chips`)
Functionally the richest screen and it mostly works. Issues: red-flag banner (G-6);
dual chips same-color (G-6); native `<select>` for reassign next to a styled button;
"Runs / Duration" k/v block misaligned; consultation rows repeat the meeting link as
plain text; Start Program/psych-override states render as plain buttons with no
moment (G-9); cycle strip is three grey rows, not a calendar strip.

### Clinician clients list (`doctor--clients-pending`)
Cards with name + "pending form" badge — reasonable bones, but no pending-work-first
sort promise visible, red-flag dot small, no cycle/next-consult info at a glance
(spec §10 wants name, age, cycle, next consult, pending badge, flag dot).

### Clinician client page (`doctor--client-overview`, `--client-form-live`)
- Overview tab is `Status: Active` — an empty room after the red-flag banner.
- The live consult form is a **5,500px single scroll** with uniform section cards, no
  sticky section nav, no in-form progress, no visible autosave (the spec's persistent
  note renders as a grey whisper at top). Submit is a floating dark pill
  bottom-right, easy to lose. Date field is a native dd/mm/yyyy input.
- Post-submit the form tab dead-ends: "Your report for this consultation has already
  been submitted." — no link to the report it just generated (brief explicitly wants
  success → report link).

### Reports (`admin--report-doctor`, `caregiver--report-nutrition`)
The best-designed surface today (own font stack, teal section headers, callout for
restrictions, k/v + tables). Gaps: assessment section is visually identical to
metadata sections (spec: assessment must lead and dominate); "Cleared with
restrictions" callout is amber here but green on the trainer tab (G-6); raw ISO
dates; k/v rows in `CONSULTATION` block waste the lead position on Date/Mode
bureaucracy; typography is system sans, not a designed document face; on-screen and
PDF share layout (good) but both lack print-grade hierarchy.

### Caregiver plans (`caregiver--plans`)
Both full plans concatenated into one giant page — a wall of k/v rows. The numbers a
family actually acts on (protein target, sessions/week, month-1 goals) are buried
mid-list. Print button exists; needs a real print stylesheet check (Phase C2).

### Onboarding wizard (`flow-04-wizard-step1`)
One §7 section per screen with progress bar and autosave — the right skeleton. But
step 1 is itself a 2,000px scroll of stacked fields (one-*section*-per-screen, not
one-thing-per-screen rhythm); no welcome step; "Saved ✓" indicator exists but is
easy to miss; Back/Next are small buttons at opposite corners; segmented options
~30px tall (G-7); completion moment not designed (submit → redirect).

### Admin overview (`admin--overview`)
Four skeletal number tiles (no deltas, no trend), an empty renewal radar, then a row
of plain buttons duplicating the nav tabs directly above them. "Consults this week:
0" while four consultations sit scheduled this week (UTC boundary — logged in
CODE-REVIEW M-3, out of design scope, but the tile's credibility suffers).

### Notifications (`admin--notifications`, `caregiver--notifications`)
Functional list with unread styling and mark-read; rows are dense text with raw
type labels; deep links work. Needs the shared status vocabulary + designed empty
state.

## 3 · Accessibility summary

| Check | Status |
|---|---|
| Body text contrast | Primary text passes; **all muted text ~3.5:1 fails AA** (G-4) |
| Focus visibility | **Fails** — 50%-alpha grey outline (G-5) |
| Touch targets | **Fails** — 32px buttons, ~30px segmented controls (G-7) |
| Base font size | 16px app / 18px portal after serif fallback — letter of §11 met, spirit (legibility) not |
| Reduced motion | No motion exists yet; no `prefers-reduced-motion` plumbing to inherit |
| Keyboard | Tab order follows DOM (fine); no skip link; board/queue not keyboard-operable; invisible focus makes it moot |
| Semantics | Headings mostly sensible; status conveyed by color alone in chips (G-6) |

## 4 · The 5 highest-impact screens to transform

1. **Caregiver portal home** — the emotional core; currently a status readout with
   zero identity. Everything the brief wants (care story, journey viz, calm cards)
   is missing. Also the base for elderly mode.
2. **Clinician client page (form tab above all)** — daily repeat work; 5,500px
   unstructured scroll with no autosave visibility, dead-end post-submit, and the
   clearance-semantics hazard. Highest friction-per-day in the product.
3. **Coordinator member page** — the operational cockpit: dual-status chips,
   scheduling, Start Program moment, red flags. Dense but currently illegible at a
   glance.
4. **Report view (screen + PDF)** — the product's tangible artifact for families and
   clinicians; already the best surface, one tier of typographic hierarchy away from
   genuinely credible medical documents.
5. **Coordinator pipeline + today queue** — the all-day tool; needs one-action rows,
   scannable columns, keyboard + Cmd+K, and working mobile.

## 5 · Dead ends & flow gaps found while walking

- Clinician form tab post-submit → text-only dead end (no report link).
- Doctor Overview tab → near-empty card ("Status: Active").
- Pipeline "Invited" card (K. V. Gopalan) → coordinator member page has no
  "resend/copy invite" affordance (admin invites page does) — a coordinator dead end.
- Caregiver home "Next consultations: none" while cycle-1 reviews aren't scheduled
  yet — true but unexplained; no "what happens next" hint (reviews are scheduled
  near cycle end).
- `share_with_caregiver` toggle absent (CODE-REVIEW H-3) — caregivers can never see
  doctor/performance reports; portal reports list quietly shows a subset with no
  explanation of what a family should expect to find there. **Design-adjacent but
  needs a new RPC → logged in DESIGN-PROPOSALS.md, not built.**

## 6 · Assumptions & constraints noted

- Demo-data advancement used only §6 RPCs as seeded users; one cosmetic date-shift
  (to show mid-cycle "Day 15") was skipped — hosted-DB writes outside RPCs are
  off-limits this session. "Day 1 of 30" states are shown instead.
- CODE-REVIEW.md HIGH/MEDIUM items (H-1 clearance wipe, S-2 caregiver suspend, M-*)
  are business/schema issues — untouched by this overhaul.
- WHO-5 trend charts (Phase C6) render only for admin/psychologist per §3; adherence
  charts from feedback data visible per existing report policies.
